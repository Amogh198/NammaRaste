Chapter 6
Results and Discussion

This chapter presents the experimental and operational results of the Bruhat Bengaluru Mahanagara Palike (BBMP) Pothole Monitoring System. The implementation validates the database schema design, system constraints, triggers, and aggregated views by linking them to functional user interfaces. The system separates user actions for standard citizens and authorized road contractors to ensure structural integrity and role-based access control. The results demonstrate how real-time geocoding triggers, auto-assignment algorithms, and Service Level Agreement (SLA) calculations operate in a live multi-user environment.

From a DBMS perspective, when a citizen submits a complaint, the database executes an upsert transaction on the citizens table before creating the complaint. It prevents duplication of citizen records by using the customer’s phone number as a unique constraint. If the citizen already exists, their primary key is retrieved; otherwise, a new record is created. The insert query executes as follows:

SQL Query - Citizen Upsert and Complaint Registration
-- Step 1: Upsert Citizen Record using unique phone number
INSERT INTO citizens (full_name, phone) 
VALUES ('John Doe', '9876543210')
ON DUPLICATE KEY UPDATE full_name = VALUES(full_name);

-- Step 2: Retrieve Citizen ID
SELECT citizen_id FROM citizens WHERE phone = '9876543210';

-- Step 3: Insert Complaint (Triggers duplicate detection automatically)
INSERT INTO complaints (citizen_id, stretch_id, complaint_lat, complaint_lng, severity, description, status)
VALUES (1, 3, 12.971598, 77.594562, 4, 'Deep pothole near central division crossing', 'open');

To ensure high data quality and avoid duplicate maintenance logs, the complaints table implements a specialized spatial trigger named flag_repeat_complaint. Before committing a new complaint, the trigger calculates the geodesic distance between the incoming coordinate pair and all active, unresolved potholes on that specific road stretch using the Spherical Law of Cosines. If a matching active complaint is found within a 100-meter radius, the system automatically flags the new record as a repeat complaint (is_repeat = 1), notifying contractors and avoiding redundant dispatch. The trigger definition is shown below:

MySQL Trigger - flag_repeat_complaint (BEFORE INSERT)
DELIMITER //
CREATE TRIGGER flag_repeat_complaint 
BEFORE INSERT ON complaints
FOR EACH ROW
BEGIN
    DECLARE nearby_count INT;

    SELECT COUNT(*) INTO nearby_count
    FROM complaints
    WHERE stretch_id = NEW.stretch_id
      AND status != 'resolved'
      AND (
          6371000 * ACOS(
              COS(RADIANS(NEW.complaint_lat)) * COS(RADIANS(complaint_lat)) *
              COS(RADIANS(complaint_lng) - RADIANS(NEW.complaint_lng)) +
              SIN(RADIANS(NEW.complaint_lat)) * SIN(RADIANS(complaint_lat))
          )
      ) <= 100;

    IF nearby_count > 0 THEN
        SET NEW.is_repeat = 1;
    END IF;
END //
DELIMITER ;

The data rendered in the complaints directory is fetched via an optimized multi-table join that aggregates citizen details, stretch coordinates, administrative ward boundaries, contractor assignments, and resolution deadlines into a single, cohesive view. The SQL query joining these relationships is structured as follows:

SQL Query - Aggregated Complaints Fetch
SELECT 
    c.complaint_id,
    ci.full_name AS citizen_name,
    rs.stretch_name,
    w.ward_name,
    c.severity,
    c.status,
    c.is_repeat,
    c.description,
    c.complaint_lat,
    c.complaint_lng,
    c.filed_at,
    con.contractor_name,
    ra.sla_deadline,
    ra.resolved_at
FROM complaints c
JOIN citizens ci ON c.citizen_id = ci.citizen_id
JOIN road_stretches rs ON c.stretch_id = rs.stretch_id
JOIN wards w ON rs.ward_id = w.ward_id
LEFT JOIN repair_assignments ra ON c.complaint_id = ra.complaint_id
LEFT JOIN contractors con ON ra.contractor_id = con.contractor_id
ORDER BY c.filed_at DESC;

To reduce backend computing overhead, these metrics are powered directly by pre-compiled database views rather than conducting heavy on-the-fly table scans. The contractor_performance view groups repair logs by contracting firm, calculating the average hours taken to shift status to 'resolved' and tallying occurrences where the current date has surpassed the pre-calculated SLA deadline without a recorded resolution. The view definition is established as follows:

SQL View - Contractor Performance Compiler
CREATE VIEW contractor_performance AS 
SELECT 
    con.contractor_name,
    COUNT(ra.assignment_id) AS total_assigned,
    AVG(TIMESTAMPDIFF(HOUR, ra.assigned_at, ra.resolved_at)) AS avg_resolution_hours,
    SUM(CASE WHEN (ra.resolved_at IS NULL AND NOW() > ra.sla_deadline) THEN 1 ELSE 0 END) AS sla_breaches
FROM repair_assignments ra 
JOIN contractors con ON ra.contractor_id = con.contractor_id
GROUP BY con.contractor_name;

Additionally, the ward_open_complaints view is utilized to display active workloads per administrative ward, allowing city planners to reallocate contractor resources to areas showing excessive backlog. The view aggregates active complaints using the following logic:

SQL View - Ward Open Workload Aggregator
CREATE VIEW ward_open_complaints AS 
SELECT 
    w.ward_name,
    COUNT(c.complaint_id) AS open_count
FROM complaints c 
JOIN road_stretches rs ON c.stretch_id = rs.stretch_id
JOIN wards w ON rs.ward_id = w.ward_id
WHERE c.status != 'resolved'
GROUP BY w.ward_name;

To populate the contractor panel, the system executes a targeted query to fetch active complaints assigned to the authenticated contractor. A left-join is performed on the repair_assignments table, filtering for records belonging to the contractor's ID. The SQL query is shown below:

SQL Query - Contractor Active Assignment List
SELECT 
    c.complaint_id,
    ci.full_name AS citizen_name,
    rs.stretch_name,
    w.ward_name,
    c.severity,
    c.status,
    c.is_repeat,
    c.complaint_lat,
    c.complaint_lng,
    c.filed_at,
    ra.sla_deadline
FROM complaints c
JOIN citizens ci ON c.citizen_id = ci.citizen_id
JOIN road_stretches rs ON c.stretch_id = rs.stretch_id
JOIN wards w ON rs.ward_id = w.ward_id
JOIN repair_assignments ra ON c.complaint_id = ra.complaint_id
WHERE ra.contractor_id = 2 -- Dynamic Contractor Session ID
  AND c.status != 'resolved'
ORDER BY c.filed_at DESC;

The real-time alerts displayed at the top of the contractor board are generated by two separate SQL queries. The first query searches for unresolved complaints marked as repeat reports (is_repeat = 1) on stretches under the contractor's jurisdiction. The second query identifies any outstanding assignments where the current timestamp has surpassed the SLA deadline without a resolution. The queries are defined as follows:

SQL Queries - Contractor Specific Alert Pulls
-- Query A: Fetch Unresolved Repeat Reports
SELECT c.complaint_id, rs.stretch_name, c.is_repeat
FROM complaints c
JOIN road_stretches rs ON c.stretch_id = rs.stretch_id
JOIN repair_assignments ra ON c.complaint_id = ra.complaint_id
WHERE ra.contractor_id = 2 
  AND c.is_repeat = 1 
  AND c.status != 'resolved';

-- Query B: Fetch Outstanding SLA Breaches
SELECT c.complaint_id, rs.stretch_name, ra.sla_deadline
FROM complaints c
JOIN repair_assignments ra ON c.complaint_id = ra.complaint_id
JOIN road_stretches rs ON c.stretch_id = rs.stretch_id
WHERE ra.contractor_id = 2 
  AND ra.sla_deadline < NOW() 
  AND c.status != 'resolved';

To enforce contract compliance, the database utilizes a trigger called check_sla_breach. This trigger is activated after any update in the repair_assignments table. If the repair remains unresolved past its SLA deadline, the trigger automatically logs the breach in the sla_breach_log table for auditing. The trigger definition is shown below:

MySQL Trigger - check_sla_breach (AFTER UPDATE)
DELIMITER //
CREATE TRIGGER check_sla_breach 
AFTER UPDATE ON repair_assignments
FOR EACH ROW
BEGIN
    IF NEW.resolved_at IS NULL AND NOW() > NEW.sla_deadline THEN
        INSERT INTO sla_breach_log (
            assignment_id, complaint_id, contractor_id, sla_deadline
        )
        SELECT NEW.assignment_id, NEW.complaint_id, NEW.contractor_id, NEW.sla_deadline
        WHERE NOT EXISTS (
            SELECT 1 FROM sla_breach_log
            WHERE assignment_id = NEW.assignment_id
        );
    END IF;
END //
DELIMITER ;

The Resolved History tab provides a historical record of all successfully closed repairs executed by the contractor. It acts as an audit log, detailing when each complaint was filed, its severity, the SLA deadline, and the resolution status. When a contractor marks a complaint as 'resolved', the database logs the completion timestamp. This removes the complaint from active alert triggers, updates metrics across views, and records the resolution time for future performance reviews.

Shifting a complaint's status to 'resolved' triggers a multi-table database transaction. The system first updates the status column in the complaints table and then sets the resolved_at column to the current timestamp (NOW()) in the repair_assignments table. The transaction statements execute as follows:

SQL Transactions - Complaint Resolution Commits
-- Step 1: Update Complaint Status to Resolved
UPDATE complaints 
SET status = 'resolved' 
WHERE complaint_id = 4;

-- Step 2: Log Resolution Timestamp in Assignments
UPDATE repair_assignments 
SET resolved_at = NOW() 
WHERE complaint_id = 4;

6.7 Summary of Operational Database Verifications

The operational tests demonstrate that the BBMP Pothole Monitoring System functions as a robust database-driven web application. By binding MySQL constraints, triggers, and views directly to frontend API endpoints, the application ensures high data integrity and performance. Standard citizens successfully file reports, which are geocoded and assigned to contractors based on ward associations, while contractors manage tasks and resolve complaints securely within their assigned jurisdictions. The results confirm that all database engines, relational schemas, constraints, and triggers operate as intended under standard operational workloads.
